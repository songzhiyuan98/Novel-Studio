## 1. Product Vision

### Product Positioning

Novel Studio is NOT an "AI novel generator." It is an **AI-powered serial fiction workbench**.

> You don't generate chapters. You run a book.
> You provide direction and decisions. AI handles execution.
> Your canon database remembers every character, every hook, every rule вҖ” across hundreds of chapters.

### Core Selling Point

AI-assisted serial fiction that maintains consistency across hundreds to thousands of chapters. The system remembers what the user forgets.

### Target Format (configurable per project)

Default (Chinese web novel): 2000-3000 characters/chapter, ~100 chapters/volume.
All format parameters are configurable via ProjectTemplate вҖ” chapter length, volume size, language, etc.

### Target Users

- Idea-rich but structure-poor creators who prefer decision-making over drafting
- Web novel / serialized fiction creators needing daily output
- Anyone who has "a great story idea" but can't sustain writing it

### Success Criteria

- MVP: 5 sequential chapters with preserved consistency
- Post-MVP: 10+ chapters
- Long-term: hundreds of chapters without degradation

## 2. Architecture Overview

### Design Principle: Separate Conversation from Orchestration

- **Chat Agent (LLM)** вҖ” understands user intent, generates user-facing responses, streams output
- **Orchestrator (deterministic code)** вҖ” workflow state machine, packet compilation, task dispatch, canon gate, impact analysis

## 3. Dynamic Schema: ProjectTemplate

### Problem

Hardcoded fields (like `realm`, `techniques`, `intimacy_score`) only work for cultivation novels. A billionaire novel needs `net_worth`, `companies`. A mystery needs `clue_inventory`, `suspicion_level`.

### Solution: All genre-specific schemas are driven by ProjectTemplate

ProjectTemplate is created at project setup. The Planner (LLM) suggests dimensions based on genre, user confirms and can customize.

```json
{
  "genre": { "primary": "зҺ„е№»", "tags": ["еҚҮзә§жөҒ", "зҲҪж–Ү"] },

  "format": {
    "chapter_length": { "min": 2000, "max": 3000, "unit": "chars" },
    "volume_size": 100,
    "language": "zh-CN"
  },

  "character_dimensions": [
    { "key": "realm", "label": "еўғз•Ң", "type": "string" },
    { "key": "techniques", "label": "жҠҖиғҪ", "type": "list" },
    { "key": "combat_style", "label": "жҲҳж–—йЈҺж ј", "type": "string" }
  ],

  "relationship_dimensions": [
    { "key": "trust", "label": "дҝЎд»»", "type": "number" },
    { "key": "tension", "label": "еј еҠӣ", "type": "number" }
  ],

  "style_profile": {
    "base": { "pov": "third_limited", "tense": "past", "prose_density": "light" },
    "custom": [
      { "key": "зҲҪзӮ№еҜҶеәҰ", "value": "extreme" },
      { "key": "жү“и„ёеұӮж¬Ў", "value": "дёүеұӮйҖ’иҝӣ" }
    ]
  },

  "writing_rules": [
    "жҜҸ500еӯ—иҮіе°‘дёҖдёӘеҶІзӘҒ",

### What This Makes Dynamic

| Component | Before (hardcoded) | After (ProjectTemplate-driven) |
|-----------|-------------------|-------------------------------|
| Character card fields | `realm`, `techniques` | Any dimensions defined in template |
| Relationship scores | `trust`, `tension`, `intimacy` | Any dimensions defined in template |
| Style profile | Fixed 9 fields | Base universal fields + custom per genre |
| Chapter length | 2000-3000 chars | Configurable per project |
| Volume boundary | ~100 chapters | Configurable per project |
| QA checks | Fixed 5 types | Base 5 + custom per genre |
| Writing rules | None | User-defined per project |

## 4. Character Card System

### Character Tiers

| Tier | Display | Lifecycle | Card Detail |
|------|---------|-----------|-------------|
| Core | Always visible in panel | Permanent | Full card |
| Important | Visible during relevant arc | Auto-archive when absent 10+ chapters, can recall | Full card |
| Episodic | Only in blueprint of appearance chapter | Auto-archive after chapter | Brief record only |

```

Dimension fields come from `ProjectTemplate.character_dimensions`. Different genre = different dimensions.

### Character Lifecycle

- New character mentioned вҶ’ system asks tier (or auto-classifies)
- Core/Important: generate full card, user confirms
- Episodic: brief record, auto-archive after chapter
- Arc ends (10+ chapters absent) вҶ’ system suggests archiving
- User can recall archived characters anytime
- Characters marked "жҡӮзҰ»" get periodic reminders: "want to bring them back?"

### Character Cards in the Workflow

1. **Planner** reads relevant character cards to suggest blueprints
2. **Writer** receives character cards in packet вҖ” must write consistently
3. **QA** checks character behavior against card (traits, speech style, taboos)
4. **Canonize** triggers automatic character state updates from Summarizer output
5. **User** can view/edit cards anytime; edits to confirmed cards trigger impact analysis

## 5. Blueprint System (Detailed Scene Cards)

### Problem with Simple Scene Cards

"жһ—еҮЎеңЁжҜ”жӯҰдёӯеҸҚеҮ»жһ—жө©еӨ©" gives Writer too much freedom вҶ’ combat without techniques, generic dialogue, literary endings.

### Solution: Detailed Writing Blueprints

Scene cards are expanded into **writing blueprints** that specify combat, dialogue, and reversal design. User confirms every detail before Writer executes.

```
Blueprint for Chapter 1, Scene 3:

Characters: жһ—еҮЎ, жһ—жө©еӨ©, жһ—иҖҒдёү(ж—Ғи§Ӯ)
Word range: 1300-2200

Combat design:
  - жһ—жө©еӨ© uses: зўҺеІ©жҺҢ (й»„зә§дёҠе“Ғ, зҒөеҠӣеҢ–еІ©зҹіиҙЁж„ҹ)
  - жһ—еҮЎ receives: еҷ¬йӯӮзҲӘ (йӯ”еёқдёҙж—¶з»ҷ, дә”жҢҮеҢ–й»‘ж°”еҗһеҷ¬зҒөеҠӣ)

Three-layer reversal:
  Layer 1 (surprise): жһ—еҮЎжҢЎдҪҸзўҺеІ©жҺҢ
  Layer 2 (face-slap): еҷ¬йӯӮзҲӘзўҺйӘЁ
  Layer 3 (aftermath): еҸ°иҜҚ + зӣ®ж Үе®ЈиЁҖ

Key dialogue (verbatim or near-verbatim):
  жһ—жө©еӨ©: "дҪ зҲ№еҪ“е№ҙеҒ·дәҶз§ҳиҚҜжүҚиў«иө¶иө°зҡ„"
  жһ—еҮЎ: "жҲ‘зҲ№жҳҜдёҚжҳҜиҙјпјҢдёүдёӘжңҲеҗҺжҲ‘иҮӘе·ұжҹҘ"

NPC reactions:
  жһ—иҖҒдёү: зһіеӯ”еҫ®зј© "иҝҷдёҖжӢӣвҖҰвҖҰжҖҺд№ҲеҸҜиғҪвҖҰвҖҰ"(дёҚиҜҙе®Ң)
  зҺӢиғ–еӯҗ: з¬‘е®№еғөдҪҸ

Forbidden in this scene:
  - Generic "е…ЁеңәйңҮжғҠ"
  - Combat without technique names
```

### Two Modes for Blueprint Creation

**Mode A: User provides outline** вҶ’ Planner expands with canon-aware details (suggests techniques, checks hooks, warns about conflicts)

**Mode B: User asks for brainstorm** вҶ’ Planner generates 2-3 directions вҶ’ user picks one вҶ’ Planner expands into blueprint

Both modes end at the same point: user confirms detailed blueprint вҶ’ Writer executes.

### Writer's Role Changes

Writer does NOT make content decisions. Writer only handles prose expression. The blueprint is the contract.

## 6. Reverse Change & Impact Analysis

### Change Risk Levels

| Level | Example | Handling |
|-------|---------|---------|
| Green (text swap) | Rename character, change term | Auto-replace + user confirm |
| Yellow (attribute tweak) | Change personality trait, adjust relationship | Show impact scope + user picks approach |
| Red (structural change) | Change character alignment, alter world rules | Full scan вҶ’ list all conflicts вҶ’ user resolves each |

### Impact Analysis Flow

```
User requests setting change
         в”Ӯ
         в–ј
Orchestrator scans all confirmed chapters + canon
         в”Ӯ
         в–ј
Generates impact report:
  - Which chapters are affected
  - Which character cards conflict
  - Which hooks/threads break
  - Severity per conflict
         в”Ӯ
         в–ј
Chat Agent presents to user with resolution options:
  A. Apply change + auto-fix (if possible)
  B. Apply from current chapter only
  C. Full rewrite of affected sections
  D. Cancel
```

### Conflict Resolution Options

For Red-level changes, system suggests narrative-compatible alternatives:
- "жһ—иҖҒдёүеҸҳеқҸдәә" вҶ’ suggest "иЎЁйқўдҝқжҠӨе®һеҲҷзӣ‘и§Ҷ" (minimizes retroactive conflicts)
- "еҸ–ж¶Ҳдҝ®зӮјдҪ“зі»" вҶ’ this breaks everything, strongly warn user

User always has override: "I know it breaks things, do it anyway" вҶ’ system marks affected chapters as "needs review."

## 7. Per-Scene Rewriting & Streaming

### Per-Scene Granularity

A chapter = [Scene 1] + [Scene 2] + [Scene 3] + [Scene 4]

When user says "scene 3 is bad":
- Only re-generate scene 3 (saves tokens)
- Maintain scene 1, 2, 4 as-is
- Writer packet includes surrounding scene text for continuity

### Streaming Output

All LLM responses stream to the user character-by-character (via Vercel AI SDK).

During generation, user can:
- **Pause** вҖ” stop generation, resume later
- **Stop & Edit** вҖ” keep generated text so far, manually edit, then continue or re-generate from edit point
- **Skip scene** вҖ” jump to next scene, mark current as "TODO"

### Mid-Chapter Iteration

```
User is discussing chapter 5 blueprint
  вҶ’ Confirms and starts writing
  вҶ’ Sees scene 2 streaming, doesn't like the dialogue
  вҶ’ Clicks "Stop"
  вҶ’ Says "иҝҷж®өеҜ№иҜқеӨӘж— иҒҠпјҢи®©иӢҸйӣЁжҷҙиҜҙдёҖеҸҘжҜ’иҲҢзҡ„"
  вҶ’ System updates blueprint for scene 2
  вҶ’ Re-generates scene 2 only
  вҶ’ Continues to scene 3
```
