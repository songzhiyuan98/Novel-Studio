# 工作流阶段（Workflow Phases）

本系统的工作流映射自"人类作者真实写作流程"：

> 灵感 → 选方向 → 设定最小规则 → 人物动机 → 结构节点 → 分镜 → 正文 → 编辑 → 修订 → 连载维护

每阶段产出工件，用户按需确认。

---

## Phase 0：定位与可写性评估

### 目标
确保材料足够"开写"，而不是直接编剧情。

### 流程

```
用户自由表达（想写什么感觉、题材、雷点、金手指大概）
                    │
                    ▼
            ┌───────────────┐
            │ Producer 解析  │
            │ 结构化约束     │
            └───────┬───────┘
                    │
                    ▼
            ┌───────────────┐
            │  Evaluator    │
            │ 输出Coverage  │
            │     Map       │
            └───────┬───────┘
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
   达到可写阈值              未达到阈值
   进入Phase 1          Producer发起最小问题集
                        （2-3个选择题）
```

### Producer 解析输出

| 类型 | 说明 |
|------|------|
| Hard constraints | 必须遵守（题材、金手指类型、禁区） |
| Soft preferences | 尽量满足（节奏偏好、感情线比例） |
| Do-not / Taboo | 绝不出现（雷点、禁止元素） |
| Unknowns | 未知项，后续通过选项解决 |

### 产物

- `StoryBible (minimal)`
- `Preferences`
- `Taboo`

## Phase 1：发散 → 收敛（方案选择）

### 目标
从无限可能收敛到一个可执行方向。

### 流程

```
            ┌───────────────────────────┐
            │ Divergent / Romance Agent │
            │   产出 A/B/C 主线方案      │
            └───────────┬───────────────┘
                        │
                        ▼
            ┌───────────────────────────┐
            │ Plot/World/Character      │
            │   给"落地性+风险"评估      │
            └───────────┬───────────────┘
                        │
                        ▼
            ┌───────────────────────────┐
            │      Producer 汇总         │
            │  方案卡 + 可调参数          │
            │  甜度/节奏/开挂强度/反转频率 │
            └───────────┬───────────────┘
                        │
                        ▼
                ┌───────────────┐
                │   用户动作     │
                │ 选方向 + 调参  │
                │ + Confirm     │
                └───────────────┘
```

### 方案卡结构

```json
{
  "option_id": "A",
  "title": "校园创业流",
  "premise": "从校园服务起步，逐步扩展到互联网",
  "risk_level": "low",
  "hook_density": "medium",
  "romance_weight": 0.3,
  "adjustable_params": {
    "pacing": ["slow_burn", "medium", "fast"],
    "power_level": ["realistic", "moderate", "op"],
    "romance_intensity": [0.1, 0.3, 0.5, 0.7]
  }
}
```

### 产物

- `Premise Confirmed`
- `PowerSystem Confirmed`
- `Cast Seed Confirmed`（部分）

## Phase 2：结构（关键节点）

### 目标
规划第一卷的关键节点，而不是详细大纲。

### 关键节点类型

| 节点 | 说明 |
|------|------|
| Inciting | 引爆点：打破日常的事件 |
| Cost | 第一次代价：证明这不是轻松游戏 |
| Mid Twist | 中段反转：改变局势的意外 |
| Darkest | 最黑暗时刻：主角最大危机 |
| Final | 终局：阶段性胜利与代价 |

### 流程

```
            ┌───────────────────────────┐
            │     Plot Architect        │
            │ 产出第一卷关键节点         │
            └───────────┬───────────────┘
                        │
                        ▼
            ┌───────────────────────────┐
            │     Producer + QA         │
            │ 检查一致性与爽点密度        │
            └───────────┬───────────────┘
                        │
                        ▼
            ┌───────────────────────────┐
            │      用户 Confirm          │
            │ 第一卷目标/终局代价的大方向  │
            │   （不逼写细节）           │
            └───────────────────────────┘
```

### 产物

- `ArcBeats`
- `Outline (draft)`

## Phase 3：分镜（场景卡）

### 目标
把每章拆解为可执行的场景卡。

### 场景卡结构

```json
{
  "scene_id": "ch1_s3",
  "chapter": 1,
  "order": 3,
  "purpose": "揭示富二代身份但钱不在自己手里",
  "conflict": "主角以为能直接拿钱，现实是父亲掌控",
  "characters": ["protagonist", "father"],
  "location": "家里/书房",
  "end_hook": "父亲提出条件：想拿钱，必须证明自己"
}
```

### 流程

```
            ┌───────────────────────────┐
            │     Scene Designer        │
            │  每章拆成 3-6 个场景卡     │
            └───────────┬───────────────┘
                        │
                        ▼
            ┌───────────────────────────┐
            │         QA 审核           │
            │   防拖沓、防无冲突         │
            └───────────┬───────────────┘
                        │
                        ▼
                  SceneCards (Reviewed)
```

### QA 检查项

- [ ] 每个场景有明确目的
- [ ] 每个场景有冲突或张力
- [ ] 每章结尾有钩子
- [ ] 场景数量适中（3-6个）
- [ ] 节奏不拖沓

### 产物

- `SceneCards (Reviewed)`

## Phase 4：正文（写章节）

### 目标
根据场景卡生成高质量正文。

### Writer 输入（Context Packet）

Writer 只读简报包，不读完整历史：

| 字段 | 内容 |
|------|------|
| Canon Snapshot | 相关正史事实（Bible 摘要） |
| Recent Summary | 最近 2-3 章摘要 + 必要少量原文引用 |
| SceneCards | 本章场景卡 |
| Constraints | 禁止项（不写系统、不神豪、不工具人等） |
| Style Guide | 文风指引 |

### 流程

```
            ┌───────────────────────────┐
            │         Writer            │
            │     生成 Chapter Draft    │
            └───────────┬───────────────┘
                        │
                        ▼
            ┌───────────────────────────┐
            │    QA（Editor+Continuity） │
            │   打分、出问题包或建议      │
            └───────────┬───────────────┘
                        │
            ┌───────────┴───────────┐
            ▼                       ▼
         不达标                   达标
    局部重写优先              进入 Reviewed
   （只重写场景3/某段）              │
            │                       │
            └───────────┬───────────┘
                        ▼
            ┌───────────────────────────┐
            │        用户动作            │
            │   Edit/Rewrite → Confirm  │
            └───────────────────────────┘
```

### QA 评估维度

| 维度 | 说明 |
|------|------|
| 节奏 | 是否拖沓、是否太赶 |
| 爽点密度 | 是否有钩子、是否有满足感 |
| 文风一致性 | 是否符合 Style Guide |
| 设定一致性 | 是否违反 Canon |
| 人物一致性 | 角色行为是否符合人设 |

### 产物

- `Chapter Confirmed`
- `Chapter Summary`
- `Bible Patch`（关系进度/新事实）

## Phase 5：连载维护（版本管理）

### 目标
保持长期连载的一致性与可维护性。

### 每次 Confirm 自动执行

```
Chapter Confirmed
        │
        ├──► 更新 Timeline（时间线）
        ├──► 更新 Promises（伏笔表）
        ├──► 更新 Romance Progress（关系进度）
        ├──► 生成 "Previously on..." 摘要
        └──► Continuity 检查潜在冲突
```

### 修改旧章流程

```
用户要求改旧章
        │
        ▼
创建 Revision（新版本）
        │
        ▼
重算受影响的摘要与一致性
        │
        ▼
标记潜在冲突章节
        │
        ▼
用户确认 Revision
```

### 产物

- 更新的 `Timeline`
- 更新的 `Promises`
- 更新的 `Romance Progress`
- `Previously On...` 摘要
- 潜在冲突报告
